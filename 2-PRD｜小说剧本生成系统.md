## 1. 项目背景与立项动机

### 1.1 背景问题

小说文本具备以下特征：

- 描述性强、镜头感弱
- 人名/称谓多样（他/她/某人/外号）
- 对话与心理描写混杂
- 时间线、场景切换隐含在叙述中

**直接用小说去做分镜或画面生成，几乎必然失败。**

------

### 1.2 项目定位

本项目的目标不是“写得好看”，而是：

> **把小说，翻译成一份「导演、演员、分镜系统都能直接使用的结构化剧本」。**

核心关键词是：
 **结构化 / 可拆解 / 可校验 / 可版本化**

------

## 2. 项目目标（Goals）

### 2.1 核心目标

1. 将小说文本转化为 **结构化剧本数据**
2. 明确场次、人物、对白、旁白、情绪
3. 支持多版本剧本生成（不同节奏 / 风格）
4. 为后续模块提供 **稳定、可复现的输入**

------

### 2.2 成功指标（KPI）

| 指标       | 目标                         |
| ---------- | ---------------------------- |
| 场次完整率 | ≥ 95% 场次具备起因-冲突-结果 |
| 对白可用率 | ≥ 90% 对白无需人工重写       |
| 角色一致性 | 同角色称谓统一率 ≥ 99%       |
| 剧本返工率 | < 20% 需要重生成             |

------

## 6. 详细功能需求

------

### 6.1 小说导入模块

#### 6.1.1 导入方式

支持：

- 文本粘贴
- 文件上传（txt / docx / md）
- URL（爬取正文，后续可选）

#### 6.1.2 章节管理

- 支持自动章节识别（第X章）
- 支持手动选择“本次生成范围”
- 支持跨章节合并为一集

**输出**

- RawChapterText（按章节）

------

### 6.2 文本清洗与标准化

#### 6.2.1 清洗规则

- 去广告段落
- 去作者注释
- 去空行、重复符号
- 标点统一（全角/半角）

------

#### 6.2.2 人名与称谓归一（关键）

**问题**

> “林凡 / 阿凡 / 他 / 少年”

**解决**

- 自动抽取角色候选
- 建立 `CharacterAliasMap`
- 后续解析统一使用 `character_id`

------

### 6.3 章节 / 事件切分

#### 6.3.1 切分策略

- 优先按章节
- 章节过长时 → 按事件点切分
- 单集目标长度受 StyleProfile 约束

------

#### 6.3.2 输出结构

```
{
  "segments": [
    {
      "segmentId": "seg-1",
      "textRange": "...",
      "summary": "主角遭遇冲突"
    }
  ]
}
```

------

### 6.4 剧情结构化（Scene）

#### 6.4.1 Scene 定义

一个 Scene 必须包含：

- 地点（明确 / 模糊）
- 时间（白天 / 夜晚 / 回忆）
- 出场人物
- 核心事件
- 情绪基调

------

#### 6.4.2 Scene 输出结构（示例）

```
{
  "sceneId": "scene-03",
  "location": "街道",
  "time": "夜晚",
  "characters": ["char_01", "char_02"],
  "conflict": "追逐",
  "turningPoint": "主角被拦下",
  "emotion": "紧张"
}
```

------

### 6.5 对白与旁白提取

#### 6.5.1 对白解析

每一句对白需明确：

- 说话人
- 对象（可选）
- 情绪
- 是否为内心独白

```
{
  "lineId": "dlg-12",
  "speaker": "char_01",
  "text": "你以为你逃得掉吗？",
  "emotion": "愤怒",
  "type": "dialogue"
}
```

------

#### 6.5.2 旁白解析

- 描述环境
- 补充心理活动
- 后续可用于 TTS

------

### 6.6 剧本版本管理（非常重要）

#### 6.6.1 多版本生成

支持策略：

- 快节奏版
- 情绪强化版
- 悬疑强化版

每次生成：

- 新 version
- 不覆盖旧版本

------

#### 6.6.2 版本状态

| 状态     | 说明     |
| -------- | -------- |
| draft    | 初稿     |
| reviewed | 人工调整 |
| locked   | 主线版本 |
| archived | 废弃     |

------

## 7. 剧本输出定义（核心产物）

### 7.1 Script.json 结构

```
{
  "scriptId": "script_v1",
  "scenes": [],
  "dialogues": [],
  "narrations": [],
  "characters_mentioned": [],
  "styleProfileRef": "style_v1",
  "meta": {
    "pace": "fast",
    "estimatedDuration": 60
  }
}
```

------

### 7.2 ScriptSummary（给人看的）

- 整体剧情摘要
- 角色关系说明
- 本集看点

------

## 8. 关键校验规则

### 8.1 自动校验

- Scene 是否完整
- 对白是否超长
- 情绪字段是否缺失
- 同一角色称谓是否统一

------

### 8.2 阻断规则

- 无 Scene → 不可进入下一项目
- 对白未绑定角色 → 标红
- 超出目标时长 → 提示重生成

------

## 9. 人工介入点（设计为“增强而非修补”）

- 一句话改写
- 删除 / 合并 Scene
- 锁定某句对白不再被 AI 改写
- 选择“主版本”

------

## 13. 当前实现范围与后续计划（实现对照说明）

### 13.1 已在当前产品中实现的能力（V1）

- 支持通过「文本粘贴」方式导入小说内容，并在前端展示原文输入区
- 提供后端 `/api/scripts` 接口，调用 DeepSeek 按统一 Script Schema 生成结构化剧本：
  - `scenes / dialogues / narrations / meta / characters_mentioned`
  - 以 `projectId + version` 管理多版本脚本
- 前端 Step2 页面展示并允许编辑基于 Scene 级别的剧本结构：
  - 每个场景包含地点、时间（DAY/NIGHT）、出场角色与剧情描述
  - 支持手动新增 / 删除场景
- 完成与项目工程的关联与持久化：
  - 生成的 Script 版本随项目一同保存与加载
  - 编辑后的 `scriptData`（场景列表）作为下游角色提取 / 分镜生成的输入
- 实现基础阻断规则：
  - 当 `scriptData` 为空（无任何 Scene）时，用户无法进入下一步（角色设计）
- 支持基础的剧本版本选择：
  - Step2 顶部提供「剧本版本」下拉框，可在不同生成版本间切换，并同步场景列表

### 13.2 仍保留为后续迭代的能力（暂未实现或仅由 AI 内部隐式处理）

- 文本清洗与标准化管线（广告、作者注释、标点统一等）：当前由大模型在提示词中隐式处理，尚未拆分为显式的可观测子模块
- 全量的章节 / 事件切分与 `segments` 输出结构：当前直接以整段输入为单位生成 Script.json，未单独暴露 `segments` 层
- 人名与称谓归一的 `CharacterAliasMap` 结构：目前仅在大模型内部语义归一，未输出为独立数据结构
- 更细粒度的对白 / 旁白编辑 UI：
  - 目前主要在 Scene 粗粒度层面提供编辑
  - 后续可增加对白列表、旁白列表的可视化与逐句编辑能力
- 完整的自动校验与错误高亮：
  - 暂未实现 Scene 完整性、对白超长、情绪字段缺失等自动检查
  - 仅实现“无 Scene 不可进入下一项目”的硬性阻断
- 更丰富的剧本生成模式（快节奏版 / 情绪强化版 / 悬疑强化版等）：
  - 当前后端已预留 `mode` 字段，但前端尚未提供模式选择 UI，也未根据不同模式做差异化策略

> 以上对照用于对齐「PRD 目标」与「当前可用能力」，后续迭代可在不破坏现有接口的前提下逐步补齐剩余能力。
