## 1. 项目背景与立项动机

### 1.1 背景问题

在AI内容生成系统中，特别是小说到漫剧视频的转化过程中，存在以下核心问题：

1. **缺乏统一的项目级与风格级配置**：各阶段AI输出风格不一致，不同模型"各自发挥"，难以控制，重跑成本高，无法复现结果，无法批量生产、无法规模化。

2. **小说文本直接生成视觉内容的失败率高**：小说文本描述性强、镜头感弱，人名/称谓多样，对话与心理描写混杂，时间线、场景切换隐含在叙述中，直接用小说去做分镜或画面生成，几乎必然失败。

3. **角色一致性是失败率最高的环节**：同一角色多镜头"换脸 / 换发型 / 换衣服"，人物设定复杂导致模型不稳定，没有角色资产，每一帧都是"重新发明角色"。

4. **剧本到视觉生成的鸿沟**：剧本本质上是"事件与对白的描述"，但视觉生成系统需要明确的镜头边界、画面构图、镜头时长、角色出场与动作，直接用剧本生成分镜图会导致画面信息过载、镜头节奏混乱、对白无处安放、后期剪辑无法控制时长。

### 1.2 项目定位

本项目是一个完整的AI漫剧生成流水线，将小说文本转化为可发布的漫剧视频。它的目标是：

> **把"用户的模糊意图"，通过一系列结构化的系统，转换成"所有AI模型可执行的、统一的、结构化的约束配置"，最终生成高质量的漫剧视频。**

整个系统由四个核心子系统组成，按严格顺序执行：

1. **项目&风格设定系统** - 全局配置与约束中枢（第一步，必须首先完成）
   - 产出：Style Profile（风格配置文件）
   - 后续步骤依赖此配置进行所有AI生成任务

2. **小说剧本生成系统** - 小说文本到可拍可演的结构化剧本（第二步，依赖第一步）
   - 产出：Script.json（结构化剧本）
   - 必须等待第一步的Style Profile完成后才能开始

3. **角色设计&角色资产库系统** - AI漫剧流水线的"角色一致性中枢"（第三步，依赖第二步）
   - 产出：Character Assets（角色资产库）
   - 必须等待第二步的剧本完成后才能开始

4. **镜头级导演脚本生成系统** - 剧本到分镜图之间的"工业级翻译层"（第四步，依赖第二、三步）
   - 产出：ShotList（镜头级导演脚本）
   - 必须等待第二步剧本和第三步角色资产完成后才能开始

**重要：每个步骤必须完全完成后才能启动下一步，系统应具备流程状态检查机制。**

**流程控制原则**：
1. **严格顺序执行**：四个步骤必须按顺序执行，第一步结束才能进行第二步，第二步结束才能进行第三步，第三步结束才能进行第四步。
2. **前置条件验证**：系统在启动每个步骤前，必须验证其所有前置步骤的输出产物已就绪且状态为“已完成”。
3. **状态锁定**：步骤进行中时，其输出产物处于“锁定”状态，不可被修改，确保下游步骤输入的一致性。
4. **流程可视化**：系统应提供流程状态看板，清晰展示每个步骤的完成状态与阻塞点。

## 2. 项目目标（Goals）

### 2.1 核心目标

1. 提供标准化的项目创建与管理能力
2. 提供统一、可复用、可版本化的风格描述体系（Style Profile）
3. 将小说文本转化为结构化剧本数据，明确场次、人物、对白、旁白、情绪
4. 从剧本中自动抽取角色并分类，为每个角色生成标准化角色设定卡和可复用、可约束的一致性资产
5. 将剧本拆解为镜头级导演脚本，为每个镜头生成完整的拍摄参数
6. 为后续所有AI项目提供唯一可信的全局配置来源（Single Source of Truth）
7. 为下游模块提供稳定、可复现的输入

### 2.2 成功指标（KPI）

| 指标           | 目标                                                                 |
| -------------- | -------------------------------------------------------------------- |
| 风格一致性     | 同一项目内多阶段输出风格一致                                         |
| 复现能力       | 同一配置多次运行结果稳定                                             |
| 下游引用成功率 | 下游模块读取 StyleProfile 成功率 ≥ 99%                                |
| 项目创建时长   | 用户完成一次项目配置 ≤ 3 分钟                                        |
| 场次完整率     | ≥ 95% 场次具备起因-冲突-结果                                          |
| 对白可用率     | ≥ 90% 对白无需人工重写                                               |
| 角色一致性     | 同一角色跨 10 个镜头外观一致                                          |
| 镜头可执行率   | ≥ 95% 镜头可直接生成画面                                             |
| 时长命中率     | 单集时长偏差 ≤ ±10%                                                  |
| 对白覆盖率     | 100% 对白绑定镜头                                                    |

### 3.1 目标用户

| 用户类型   | 需求                                                                 |
| ---------- | -------------------------------------------------------------------- |
| 内容创作者 | 快速配置一个"能用的项目"，把小说快速转成视频内容，快速得到"稳定能用"的角色 |
| 工作室/MCN | 批量生产，风格统一，降低重生成成本                                   |
| AI 运营    | 控制模型输出，减少失败，批量生成、挑版本                             |

| 导演 / 策划 | 快速生成镜头方案，控制节奏与高潮                                      |
| 下游系统    | 自动读取剧本结构，稳定引用角色，直接消费 ShotList                    |

### 3.2 核心使用场景

- 创建一个新漫剧项目
- 为同一小说生成多个风格版本
- 上传小说章节，生成单集剧本
- 同一小说生成多个节奏版本
- 从剧本一键生成角色资产
- 人工挑选"最像"的角色版本
- 将一集剧本自动拆成 10~30 个镜头
- 为不同节奏生成不同 ShotList 版本
- 复制已有项目作为模板
- 项目中途调整风格并重跑后续流程
- 人工微调后锁定主版本
- 将剧本作为后续流水线唯一文本输入
- 锁定角色关键特征供后续镜头复用
- 在多个分镜、视频中反复调用同一角色
- 锁定关键镜头用于封面或高潮
- 将 ShotList 作为分镜/视频的输入源

## 6. 详细功能需求

### 6.1 项目&风格设定系统

#### 6.1.1 项目管理模块

##### 6.1.1.1 创建项目

**功能描述**
 用户通过点击创建按钮，弹出模态框输入项目名称后创建一个新的漫剧生成项目。

**交互流程**
1. 用户点击首页右上角或空状态下的"创建项目"按钮
2. 系统弹出创建项目模态框
3. 用户输入项目名称（必填）
4. 用户点击"创建"按钮
5. 系统创建新项目并自动进入项目编辑界面

**输入字段**

- 项目名称（必填）

**输出**

- project_id
- 项目基础元信息
- 自动导航到项目编辑界面

**业务规则**

- 项目名称唯一（同一用户下）
- 创建即生成默认 Style Profile（草稿态）
- 项目创建后自动将项目名称同步到项目内部设置

##### 6.1.1.2 项目卡片交互

**功能描述**
 优化项目卡片的交互方式，提升用户体验。

**交互流程**
1. 用户点击项目卡片的任意位置（除删除按钮外）
2. 系统直接打开该项目，进入项目编辑界面

**业务规则**
- 点击删除按钮时不会触发项目打开操作
- 卡片悬停时有视觉反馈（边框颜色变化）

##### 6.1.1.3 项目编辑界面导航

**功能描述**
 在项目编辑界面添加返回首页的便捷导航功能。

**交互流程**
1. 用户在项目编辑界面点击顶部导航栏的"返回首页"按钮
2. 系统保存当前项目状态
3. 用户被导航回首页的项目列表界面

**UI设计**
- 返回首页按钮位于顶部导航栏左侧
- 按钮包含返回箭头图标和"返回首页"文字
- 按钮样式与其他导航元素保持一致

##### 6.1.1.4 首页空状态优化

**功能描述**
 优化首页空状态的界面设计，简化操作入口。

**UI设计**
- 移除空状态中央的"新建项目"按钮
- 仅保留右上角的创建项目按钮作为唯一的创建入口
- 保持空状态的其他视觉元素（图标、文字说明）不变

##### 6.1.1.5 项目设定流程强化

**功能描述**
 确保用户必须完成所有必要的项目设定才能进入下一步流程。

**业务规则**
- 必须填写项目名称
- 必须选择视觉风格
- 必须选择画幅比例
- 只有完成以上所有设定，才能点击"确认设定并进入下一步"按钮
- 剧本生成步骤在侧边栏中会显示为禁用状态，直到完成项目设定

**交互设计**
- "确认设定并进入下一步"按钮在未完成所有设定时保持禁用状态
- 禁用状态下按钮有明确的视觉反馈（灰色、不可点击）
- 完成所有设定后按钮自动启用

##### 6.1.1.6 项目状态

| 状态     | 说明             |
| -------- | ---------------- |
| Draft    | 草稿，允许修改   |
| Locked   | 锁定，供下游使用 |
| Archived | 归档             |

#### 6.1.2 风格模板模块

##### 6.1.2.1 预设风格模板

**示例**

| 字段             | 说明           |
| ---------------- | -------------- |
| episode_duration | 单集时长（秒） |
| shots_per_minute | 镜头密度       |
| narration_ratio  | 旁白占比       |
| dialogue_ratio   | 对白占比       |
| pacing           | 快 / 中 / 慢   |

##### 6.1.3.3 AI 行为约束（高级）

| 字段              | 说明           |
| ----------------- | -------------- |
| consistency_level | 角色一致性要求 |
| creativity_level  | 创意自由度     |
| motion_intensity  | 运镜强度       |
| retry_policy      | 失败重试策略   |

#### 6.1.4 Style Profile 输出（核心产物）

##### 6.1.4.1 Style Profile 定义

**这是本项目的最终、最重要交付物**

```
{
  "projectId": "xxx",
  "version": "v1.0",
  "visual": {...},
  "narrative": {...},
  "aiConstraints": {...},
  "platform": {...}
}
```

##### 6.1.4.2 设计原则

- 完全结构化
- 无自然语言歧义
- 可直接作为 Prompt 输入
- 可持久化、可版本化

#### 6.1.5 版本管理

- 每次修改生成新版本
- 支持回滚
- 下游模块必须绑定具体版本号

#### 6.1.6 对外接口（供下游使用）

- `GET /projects/{id}/style-profile`
- `GET /projects/{id}/style-profile?version=v1.0`

**规则**

- Locked 状态才能被下游读取
- Draft 状态不可执行生成任务

### 6.2 小说剧本生成系统

#### 6.2.1 小说导入模块

##### 6.2.1.1 导入方式

支持：

- 文本粘贴
- 文件上传（txt / docx / md）
- URL（爬取正文，后续可选）

##### 6.2.1.2 章节管理

- 支持自动章节识别（第X章）
- 支持手动选择"本次生成范围"
- 支持跨章节合并为一集

**输出**

- RawChapterText（按章节）

#### 6.2.2 文本清洗与标准化

##### 6.2.2.1 清洗规则

- 去广告段落
- 去作者注释
- 去空行、重复符号
- 标点统一（全角/半角）

##### 6.2.2.2 人名与称谓归一（关键）

**问题**

> "林凡 / 阿凡 / 他 / 少年"

**解决**

- 自动抽取角色候选
- 建立 `CharacterAliasMap`
- 后续解析统一使用 `character_id`

#### 6.2.3 章节 / 事件切分

##### 6.2.3.1 切分策略

- 优先按章节
- 章节过长时 → 按事件点切分
- 单集目标长度受 StyleProfile 约束

##### 6.2.3.2 输出结构

```
{
  "segments": [
    {
      "segmentId": "seg-1",
      "textRange": "...",
      "summary": "主角遭遇冲突"
    }
  ]
}
```

#### 6.2.4 剧情结构化（Scene）

##### 6.2.4.1 Scene 定义

一个 Scene 必须包含：

- 地点（明确 / 模糊）
- 时间（白天 / 夜晚 / 回忆）
- 出场人物
- 核心事件
- 情绪基调

##### 6.2.4.2 Scene 输出结构（示例）

```
{
  "sceneId": "scene-03",
  "location": "街道",
  "time": "夜晚",
  "characters": ["char_01", "char_02"],
  "conflict": "追逐",
  "turningPoint": "主角被拦下",
  "emotion": "紧张"
}
```

#### 6.2.5 对白与旁白提取

##### 6.2.5.1 对白解析

每一句对白需明确：

- 说话人
- 对象（可选）
- 情绪
- 是否为内心独白

```
{
  "lineId": "dlg-12",
  "speaker": "char_01",
  "text": "你以为你逃得掉吗？",
  "emotion": "愤怒",
  "type": "dialogue"
}
```

##### 6.2.5.2 旁白解析

- 描述环境
- 补充心理活动
- 后续可用于 TTS

#### 6.2.6 剧本版本管理（非常重要）

##### 6.2.6.1 多版本生成

支持策略：

- 快节奏版
- 情绪强化版
- 悬疑强化版

| -------- | -------- |

##### 6.2.8.1 自动校验

- Scene 是否完整
- 对白是否超长

##### 6.2.8.2 阻断规则

### 6.4 镜头级导演脚本生成系统

#### 6.4.1 Scene → Shot 拆分模块

##### 6.4.1.1 拆分原则（非常关键）

一个 Shot 应满足：

- 单一视觉焦点
- 单一主要动作或情绪
- 可在 2–6 秒内表达清楚（可配置）

##### 6.4.1.2 拆分策略

- 动作变化 → 新镜头
- 情绪变化 → 新镜头
- 视角变化 → 新镜头
- 对白切换 → 可新镜头（视节奏）

**输出示例**

```
{
  "shotId": "shot_01",
  "sceneId": "scene_03",
  "order": 1
}
```

#### 6.4.2 镜头属性生成模块

##### 6.4.2.1 摄影属性（Camera）

| 属性       | 示例                      |
| ---------- | ------------------------- |
| shotSize   | 特写 / 近景 / 中景 / 全景 |
| cameraMove | 静止 / 推 / 拉 / 摇 / 移  |
| angle      | 平视 / 仰视 / 俯视        |

##### 6.4.2.2 构图属性（Composition）

- 角色站位（左 / 中 / 右）
- 视线方向
- 前景 / 背景层次
- 构图比例（三分法 / 居中）

##### 6.4.2.3 情绪与动作

- 角色表情
- 氛围关键词（紧张 / 温馨 / 压抑）
- 关键动作描述（简洁）

##### 6.4.2.4 时长生成

- 基于：
  - 镜头类型
  - 对白长度
  - StyleProfile 节奏
- 单镜头时长：1.5s – 8s（可配置）

#### 6.4.3 台词与旁白绑定

##### 6.4.3.1 绑定规则

- 每句对白 **必须** 绑定一个 Shot
- 可选展示方式：
  - 屏幕字幕
  - 旁白
  - 角色内心独白

**示例**

```
{
  "dialogueRef": "dlg_12",
  "display": "subtitle",
  "timing": "full"
}
```

##### 6.4.4.1 ShotList 结构定义（核心）

```
{
  "shotId": "shot_08",
  "sceneId": "scene_03",
  "order": 8,
  "characters": ["char_01", "char_02"],
  "camera": {

    "move": "static",
    "angle": "eye_level"
  },

  },

  "emotion": "紧张",
  "dialogueRef": ["dlg_12"],
  "durationSec": 3.5,

  "flags": ["key_shot"]
}
```

##### 6.4.4.2 镜头级表格输出格式（新增）

最终输出的导演脚本将以表格形式呈现，每个镜头作为一个独立的行条目：

| 镜头序号 | 镜头ID | 场景ID | 角色 | 镜头规格 | 构图 | 动作描述 | 情绪 | 对白引用 | 时长(秒) | 图片生成提示词 | 标记 |
|----------|--------|--------|------|----------|------|----------|------|----------|----------|----------------|------|
| 1 | shot_01 | scene_01 | ["char_01"] | 特写/静止/平视 | 单人/聚焦主角 | 主角惊讶 | 惊讶 | dlg_01 | 2.5 | Close-up of a surprised young man in school uniform, anime style, expressive eyes, detailed facial features | key_shot |
| 2 | shot_02 | scene_01 | ["char_01", "char_02"] | 中景/静止/平视 | 双人/主角在左 | 主角与配角对峙 | 紧张 | dlg_02-dlg_05 | 4.2 | Medium shot of two characters facing each other in school uniforms, anime style, tense atmosphere, dynamic poses | - |
| ... | ... | ... | ... | ... | ... | ... | ... | ... | ... | ... | ... |

**表格说明：**
- 每一行代表一个独立的镜头
- "图片生成提示词"列提供用于AI图像生成的具体提示词
- 所有字段都来自ShotList结构，以表格形式便于导演和制作团队查阅

| 状态     | 说明     |

## 8. 异常与边界场景

- 用户修改风格但已有任务在运行 → 新任务用新版本
- 下游请求不存在版本 → 返回错误并阻断
- 模板升级 → 不影响已有项目
- 小说文风差异极大 → 系统需具备一定的适应性
- 对话不口语化 → 系统需进行一定的标准化处理
- 多角色混乱 → 系统需进行角色归一处理
- 角色设定过复杂导致崩坏 → 系统需进行简化处理
- 情绪变化大导致脸不稳 → 系统需进行情绪稳定处理
- 路人角色过多浪费算力 → 系统需进行路人模板化处理
- 镜头颗粒度过细 → 生成成本高 → 系统需进行优化
- 镜头过粗 → 画面不可控 → 系统需进行细化处理
- 多角色场景复杂 → 系统需进行简化处理

## 11. 技术架构与系统集成

### 11.1 系统架构图

```
用户输入 → 项目&风格设定系统 → 小说剧本生成系统 → 角色设计&角色资产库系统 → 镜头级导演脚本生成系统 → 下游系统
```

### 11.2 系统集成方式

- 所有子系统通过 RESTful API 进行通信
- 每个子系统的输出作为下一个子系统的输入
- 系统具备严格的流程控制机制，确保前一步骤完成后才能启动下一步
- 所有子系统共享同一个数据库

### 11.3 数据流转

- 项目&风格设定系统输出 StyleProfile → 供后续所有步骤使用
- 小说剧本生成系统输出 Script → 供角色设计和镜头脚本步骤使用
- 角色设计&角色资产库系统输出 CharacterAssets → 供镜头脚本步骤使用
- 镜头级导演脚本生成系统输出 ShotList → 最终输出成果

### 11.4 流程依赖与状态管理

- **步骤锁定机制**：每个步骤完成后必须显式标记为"已完成"，才能解锁下一步
- **前置条件检查**：系统在启动任一步骤前，必须验证其依赖的前置步骤已完成
- **状态追踪**：实时追踪每个项目的四个步骤状态（待开始/进行中/已完成）
- **阻断机制**：如果前置步骤未完成，系统应阻止后续步骤的启动并给出明确提示
